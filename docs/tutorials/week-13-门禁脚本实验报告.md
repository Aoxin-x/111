## 三、实验内容与操作步骤

### 任务1：设计门禁检查（20分钟）

#### 1.1 设计思路

发布门禁是上线前的强制校验关卡，目的是拦截存在缺陷、不合规的代码进入生产环境，从4个维度设计检查规则：

1. **代码规范门禁**：代码格式化、注释、语法错误、高危函数检测
2. **依赖安全门禁**：第三方库漏洞、版本锁定校验
3. **配置文件门禁**：生产环境敏感配置、密钥硬编码、日志等级校验
4. **分支权限门禁**：禁止直接向main主分支推送代码、合并前必须评审

#### 1.2 输出成果

整理门禁检查架构图，划分**阻断型门禁**（不通过直接终止发布）、**告警型门禁**（仅提示不阻断）两类规则。

#### 1.3 操作步骤

1. 梳理线上发布事故案例，提取高频风险点；
2. 分类划分4大类检查维度，定义每类检查的通过标准；
3. 区分阻断/告警两种门禁等级，明确失败后的处理策略；
4. 文档化输出《门禁检查设计方案》。

#### 结果分析

分层级门禁可以精准区分严重风险与轻微规范问题，既保障生产稳定，又不会因小规范问题阻塞开发流程。

### 任务2：编写门禁脚本（30分钟）

采用Python实现自动化门禁校验脚本，核心功能：代码静态检查、配置敏感词扫描、分支校验。

#### 完整示例脚本：gate_check.py

```python
import os
import re
import subprocess

gate_block = False

def check_branch():
    res = subprocess.check_output(["git", "rev-parse", "--abbrev-ref", "HEAD"]).decode().strip()
    if res == "main":
        print("[阻断门禁] 禁止直接在main分支提交代码！")
        global gate_block
        gate_block = True

def check_sensitive_config():
    sensitive_words = ["password", "secret", "ak=", "sk=", "root@127.0.0.1"]
    for root, _, files in os.walk("."):
        for file in files:
            if file.endswith((".yml", ".json", ".ini")):
                path = os.path.join(root, file)
                with open(path, "r", encoding="utf-8", errors="ignore") as f:
                    content = f.read()
                    for word in sensitive_words:
                        if word in content:
                            print(f"[阻断门禁] 文件{path}包含敏感字段 {word}")
                            global gate_block
                            gate_block = True

def code_lint_check():
    try:
        subprocess.check_output(["python", "-m", "flake8", "."], stderr=subprocess.STDOUT)
    except subprocess.CalledProcessError as e:
        print("[告警门禁] 代码存在规范问题：", e.output.decode())

if __name__ == "__main__":
    print("===== 发布门禁自动化校验开始 =====")
    check_branch()
    check_sensitive_config()
    code_lint_check()
    print("===== 门禁校验结束 =====")
    if gate_block:
        exit(1)
    else:
        print("所有阻断门禁校验通过，可进入下一发布步骤")
        exit(0)
```

#### 执行命令

```bash
python3 gate_check.py
```

#### 执行结果

- 存在违规项：脚本输出阻断提示，程序退出码1，终止发布流水线；
- 无高危违规：仅输出代码规范告警，退出码0，放行发布流程。

#### 结果分析

脚本实现自动化前置校验，替代人工逐项检查，降低漏检风险，可直接集成CI/CD发布流水线。

### 任务3：创建检查清单（20分钟）

#### 发布门禁上线检查清单

##### 一、代码门禁检查项（阻断）

| 序号 | 检查项 | 状态 |
|------|--------|------|
| 1 | 当前分支非main/master主分支 | ✅/❌ |
| 2 | 代码无语法错误、可正常编译 | ✅/❌ |
| 3 | 无高危函数、废弃接口调用 | ✅/❌ |
| 4 | 新增代码补充单元测试 | ✅/❌ |

##### 二、配置文件门禁检查项（阻断）

| 序号 | 检查项 | 状态 |
|------|--------|------|
| 1 | 无明文密钥、账号密码硬编码 | ✅/❌ |
| 2 | 生产数据库、缓存地址使用环境变量注入 | ✅/❌ |
| 3 | 日志级别不为DEBUG（生产环境） | ✅/❌ |

##### 三、安全依赖门禁检查项（阻断）

| 序号 | 检查项 | 状态 |
|------|--------|------|
| 1 | 第三方依赖无高危漏洞 | ✅/❌ |
| 2 | 依赖版本统一锁定，无浮动版本号 | ✅/❌ |

##### 四、流程规范检查项（告警）

| 序号 | 检查项 | 状态 |
|------|--------|------|
| 1 | MR/PR已通过至少1人代码评审 | ✅/❌ |
| 2 | 更新内容填写发布变更说明 | ✅/❌ |
| 3 | 回滚方案、故障预案准备完成 | ✅/❌ |

#### 使用说明

发布前逐项核对，阻断项全部通过才可执行上线；告警项未完成需记录风险并同步运维。

## 五、实验遇到的问题与解决

| 序号 | 问题描述 | 解决方案 |
|------|---------|---------|
| 1 | 脚本遍历文件时读取特殊编码配置文件报错 | 打开文件时添加 `errors="ignore"` 参数，跳过无法解析字符 |
| 2 | 切换分支后git命令获取分支名输出乱码 | 使用 `decode().strip()` 清理命令输出换行符与空格 |
| 3 | 检查清单分不清阻断和告警项，容易漏看 | 清单使用分块区分阻断/告警，阻断项标红优先核对 |
| 4 | 脚本无法集成流水线，退出码不生效 | 明确阻断场景使用 `exit(1)`，放行使用 `exit(0)`，适配CI流程判定逻辑 |
